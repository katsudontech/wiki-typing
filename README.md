# ⌨️ WikiTyping

Wikipediaの記事をLLM（Gemini API）で動的に要約・整形し、タイピングゲームの教材として遊べるWebアプリケーションです。

## 💡 プロジェクトの目的
世の中にある固定のテキストを打つだけのタイピングゲームではなく、「自分の興味があるWikipediaの記事」を遊びながらインプットできる体験を作りたいと考え開発しました。

## 🛠 技術スタック
- **Frontend / Backend**: Next.js (TypeScript)
- **AI / LLM**: Gemini API

## 🔥 アピールポイント・こだわりの工夫点
本アプリケーションの開発で一番苦労したのは、「LLMが生成したテキストを、いかにタイピングゲームのUI（入力判定やハイライトなど）で扱いやすい形にするか」でした。

### 1. フロントエンドの描画から逆算したプロンプト設計とハイライト機能
最初は、LLMに「漢字混じりの文」と「読みがな」を単なる文字列として出力させていました。しかし、これだとフロントエンド側で「今ユーザーが打っている文字が、元の文章のどの漢字に対応しているか」を判定してハイライトさせるのが非常に難しかったです。

そこで、LLMには単なるテキストではなく、フロントエンドで扱いやすいデータ構造をそのまま出力してもらうようにプロンプトを工夫しました。

**【データ構造の工夫】**
Gemini APIに例（Few-shot）を渡し、**「文節とその読みがなのペアをJSON配列で返す」**ように指示しました。

```javascript
// 実際のプロンプトの抜粋
const prompt = `以下の日本語を、タイピング練習の文章としてふさわしい形で、最大${maxLength}文字に要約した上で、
元の文章（漢字交じり）と、その読み方（全てひらがな）を、意味のまとまりごとに分割してJSON配列で返して。

【超重要ルール】
1. 読み方（reading）側には、『』や「」、（）などの記号を絶対に含まないこと。空文字（""）にして。
2. 句読点（。、）もタイピングの邪魔になるので、読み方（reading）側からは取り除き、空文字（""）にして。
3. 全て繋げたときに元の文章になるように。

フォーマット例:
{
  "segments": [
    {"text": "吾輩", "reading": "わがはい"},
    {"text": "は", "reading": "は"},
    {"text": "猫", "reading": "ねこ"},
    {"text": "である", "reading": "である"},
    {"text": "。", "reading": ""}
  ]
}
テキスト：${promptSourceText}`;
```

**【文節単位のハイライトと自動スクロール】**
このJSONデータを使うことで、文字単位ではなく「意味のまとまり（文節）単位」で、入力に合わせてリアルタイムにハイライトする処理をシンプルに実装できました。また、長文になっても入力中の場所が常に見えるように自動スクロール（`scrollIntoView`）も入れ、快適にタイピングできるUIを作りました。

### 2. タイピングのリズムを崩さないためのノイズ除去
LLMが読みがなを正しく生成してくれても、「」や（）などの記号、句読点（。、）が含まれているとタイピングの邪魔になってしまいます。
そのため、プロンプトのルールで「読みがな（reading）には絶対に記号や句読点を含めず、空文字（""）にすること」と強めに指示し、スムーズにタイピングできるように調整しました。

### 3. Wikipedia APIを使った「無限のお題生成」
あらかじめ用意した固定の文章ではなく、WikipediaのAPI（ランダム取得や検索）を使うことで、無限に新しいお題で遊べるようにしました。
単なるタイピング練習作業にならず、「自分の好きなカテゴリの知識」や「全く知らなかったWikipediaの記事」に偶然出会えるような、遊んでいて飽きない体験を目指しました。

### 4. 外部API連携時のエラーハンドリング
Wikipedia APIとGemini APIという2つの外部APIを連続して呼んでいるため、APIのタイムアウトや、LLMが指定したJSONフォーマットから外れた出力をするリスクがありました。
そこで、データの取得や生成に失敗してもアプリが落ちないように例外処理を入れ、エラー時にはわかりやすい画面を表示して「再試行」ボタンを配置するなど、実際に使ってもらうことを想定したエラーハンドリングを行っています。

## 🚀 今後の展望・改善点

**1. バックグラウンド生成による待機時間の解消**
現在、ゲームを始めるたびに「Wikipediaの取得」と「Geminiによる生成」を待つ必要があり、少しテンポが悪いのが課題です。
これを解決するために、最初の1回目だけロード時間をかけてゲームを開始し、**ユーザーがプレイしている裏（バックグラウンド）で次のゲームのお題をあらかじめ生成しておく仕組み**を実装したいと考えています。これにより、2回目以降は待ち時間なしでサクサク遊べるようにします。

**2. APIのレートリミット対策**
現在はGemini APIの無料枠を利用していますが、ユーザーが連続してプレイするとAPIの制限（レートリミット）に引っかかってしまう可能性があります。今後は、APIリクエストの回数を最適化したり、必要に応じて有料枠へ移行するなどして、安定して遊べる環境を整えたいです。

**3. スコア機能の拡張とランキングの実装**
現在でもKPM（1分間あたりの打鍵数）や正確率、e-typing基準のスコアを計算し、ブラウザ（localStorage）に保存する機能はあります。
今後はモチベーションをより高められるよう、Supabaseなどを使ってスコアをクラウドに保存し、全国ランキングを作りたいです。また、「どのキーを打つのが苦手か（例：特定の拗音など）」を可視化してフィードバックする機能も追加したいと考えています。

## 💻 開発環境のセットアップ手順

```bash
git clone [https://github.com/YourUsername/WikiTyping.git](https://github.com/YourUsername/WikiTyping.git)

# 依存関係のインストール
npm install

# 環境変数の設定 (.env.localを作成)
# GEMINI_API_KEY=your_api_key_here を追加

# 開発サーバーの起動
npm run dev
```
